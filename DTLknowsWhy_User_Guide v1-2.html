DTLknowsWhy — User Guide v1.2

A practical guide to running DTLknowsWhy, reading its reports, and using the expert diagnosis and comparative analysis tools.

version1.0 platformWindows 10 / 11 runtimePython 3.10+ languagesfr / en licenseMIT

Overview

DTLknowsWhy is a Windows network diagnostic tool. Run it on a machine to get an instant picture of its configuration: OS version, network profile, IP settings, key Windows services, and basic connectivity. Optionally point it at another machine on the network to classify it and test its ports.

Every run produces three files: a JSON snapshot (raw data for archiving or comparison), a plain-text report (printable, shareable), and a dark-themed HTML report (for reading in a browser). Reports are available in French (default) and English.

Two additional tools work on top of the snapshots: expert.py analyses a single snapshot and issues diagnosis findings with remediation steps, and compare.py compares two snapshots from two machines to pinpoint what is different.

Quick Start

Local snapshot (French report, default)

python agent.py --snapshot

Local snapshot + remote test

python agent.py --target 192.168.1.42

English report

python agent.py --snapshot --lang en

Expert diagnosis on the latest snapshot

python expert.py

Compare two machines

python compare.py PC-A_snapshot_20260528.json PC-B_snapshot_20260528.json

All commands are run from the project root directory. Output files are written to the same directory.

Requirements

  • Windows 10 or Windows 11
  • Python 3.10 or later — standard library only, no pip install needed
  • PowerShell 5.1 or 7+ (pre-installed on all supported Windows versions)
  • Administrator rights recommended for full results (see Administrator rights)
No dependencies: DTLknowsWhy uses only the Python standard library. Copy the project folder to any Windows machine with Python installed and run it immediately.

Modes

Local snapshot

Collects a full picture of the local machine and writes three output files.
Command
python agent.py --snapshot [--lang {fr,en}]
What is collected
  • System: hostname, logged-in username, administrator status, Windows version and build number
  • Network: active profile name and category (Public / Private / Domain), IPv4 address, subnet mask, default gateway, DNS servers, DHCP status, NetBIOS status
  • Local tests: ping loopback (127.0.0.1), ping self (own IP), ping gateway
  • Windows services: status of LanmanServer, LanmanWorkstation, FDResPub, fdPHost, lmhosts
Output

Three files named HOSTNAME_snapshot_TIMESTAMP.json, HOSTNAME_report_TIMESTAMP.txt, and HOSTNAME_report_TIMESTAMP.html. A summary is printed to the console with the paths of the three files.

Remote target

Runs a local snapshot plus a battery of remote diagnostic tests against a target IP or hostname.
Command
python agent.py --target 192.168.1.42 [--lang {fr,en}]
What is tested on the target
  • Ping: basic reachability
  • Hostname resolution: DNS / NetBIOS name lookup
  • TCP port probes: ports 80 (HTTP), 139 (NetBIOS), 443 (HTTPS), 445 (SMB)
  • MAC address: looked up from the local ARP table
  • Target classification: automatic type assignment based on test results
Target classification
ClassificationBadgeMeaning
probable_windowsGreenTCP 445 or 139 open — likely a Windows workstation
probable_mobile_appleBlueHostname contains iphone or ipad
probable_mobile_androidBlueHostname contains android
probable_deviceOrangeTCP 80 or 443 open — network appliance or smart device
unknown_network_deviceOrangeMAC resolved but no recognisable open ports
unknown_hostGreyPing succeeded but nothing else resolved
unreachableRedPing failed
Output

Same three files as a local snapshot, with an additional Remote target section in both reports.

Tip: The remote test suite always runs a local snapshot first. Use --target alone — no need to also pass --snapshot.
MAC address: the MAC is read from the local ARP table and only resolves if the target is on the same network segment and has been contacted recently. Testing your own IP will show a special message instead of a MAC address.

Server mode

Starts a lightweight HTTP server on port 5050. A remote machine (e.g. NetDTL) can trigger a snapshot collection and receive the result as JSON over the network.
Command
python agent.py --listen
How it works

The server listens on all interfaces (0.0.0.0:5050). When it receives a valid request, it runs a full --snapshot collection locally, writes the three output files, and returns the JSON snapshot over HTTP.

Making a request
GET http://192.168.1.10:5050/snapshot?key=DTLSECRET&lang=fr
ParameterRequiredDescription
keyYesMust match the API_KEY constant in agent/server.py. Default value: DTLSECRET. Returns HTTP 403 if wrong.
langNofr (default) or en
Security: The server has no TLS and the API key is stored in plain text. Use only on trusted local networks. Change the default key before deploying.

Output Files

JSON snapshot

Raw structured data — all collected values in a single JSON file. Suitable for archiving, diffing, or feeding into the expert and compare tools.
Filename
HOSTNAME_snapshot_YYYYMMDD_HHMMSS.json
Top-level keys
  • system — identity and OS info
  • network — network configuration
  • tests — local ping test results
  • services — Windows service statuses
  • remote_tests — present only when --target was used

Text report

Human-readable plain text, UTF-8 encoded. Fixed-width label columns for easy reading. Sections: identity, system, network, Windows services, local tests, and (if applicable) remote target with interpretation paragraph.
Filename
HOSTNAME_report_YYYYMMDD_HHMMSS.txt

HTML report

Self-contained HTML file with a dark terminal theme. Open in any browser — no server needed. Values are colour-coded: green for positive, red for negative, grey for unknown. The Windows services and local tests sections are collapsible.
Filename
HOSTNAME_report_YYYYMMDD_HHMMSS.html
Sections
  • Header: title, generation timestamp, NetDTL logo
  • Local machine: identity, system, network, Windows services (collapsible), local tests (collapsible)
  • Remote target (if applicable): identification with classification badge, remote connectivity tests
Logo: the HTML report loads netdtl_logo.png from the same directory. Place the logo file alongside the report for it to display correctly.

Expert Tools

Expert diagnosis

Analyses a snapshot and outputs structured findings with levels and remediation steps. Designed for quick identification of network configuration issues.
Command
# Analyse the latest snapshot automatically
python expert.py

# Analyse a specific snapshot
python expert.py PC-EXAMPLE_snapshot_20260528_081930.json
Finding levels
LevelMeaning
OKNo issue detected — everything is working as expected
INFOInformational — no action required, but worth noting
WARNPotential issue — may cause problems in some scenarios
FAILConfirmed problem — likely causing the reported issue
What is analysed
  • Whether the agent ran as administrator
  • Gateway reachability (local connectivity)
  • Network profile (Public profile blocks SMB)
  • LanmanServer and LanmanWorkstation service status
  • Remote target type and SMB accessibility (if remote_tests is present)
Sample output
=== DTLknowsWhy Diagnosis ===

Machine : PC-EXAMPLE
Système : Windows 11 Pro 23H2
Profil réseau : Public
IP locale : 192.168.1.10

[WARN] Le profil réseau Public peut bloquer le partage Windows.
      Action : Passer le profil réseau en Privé.

[OK] Connectivité locale correcte (passerelle 192.168.1.1).

[FAIL] Le service LanmanServer (partage Windows) est arrêté.
      Action : Démarrer le service : sc start LanmanServer
Auto-detection: if no snapshot file is given, expert.py automatically picks the most recently modified *_snapshot_*.json file in the current directory. Run python agent.py --snapshot first, then python expert.py with no arguments for a one-step workflow.

Comparative analysis

Compares two snapshots — one from a machine that works correctly (PC A) and one from the machine under investigation (PC B) — and identifies the configuration differences that most likely explain the problem.
Command
python compare.py PC-A_snapshot_20260528.json PC-B_snapshot_20260528.json
Typical use case

PC B cannot access network shares that PC A can reach. Collect a snapshot on each machine, copy both files to the same directory, and run compare.py. The output will flag whether the issue is a Public network profile, a stopped SMB service, or a gateway connectivity problem.

Finding tags
TagSeverityMeaning
[CAUSE CERTAINE]HighA confirmed difference that directly causes the problem (e.g. SMB service stopped on PC B)
[CAUSE PROBABLE]MediumA likely cause — a configuration known to block the feature in question
[DONNÉES INSUFFISANTES]LowA field is missing or unknown on PC B — cannot draw a conclusion
[AUCUNE ANOMALIE MAJEURE]NoneNo significant difference detected between the two snapshots
Sample output
=== DTLknowsWhy Comparative Diagnosis ===

PC A : PC-ALICE
PC B : PC-BOB

[CAUSE CERTAINE]
Le partage Windows est désactivé sur PC B (LanmanServer arrêté).

[CAUSE PROBABLE]
PC B utilise un profil réseau Public, ce qui peut bloquer SMB
et la découverte réseau.
Directional: compare.py only looks for ways in which PC B differs from PC A. If both machines have the same misconfiguration, it will not be flagged. For absolute analysis, use expert.py on each snapshot individually.

Administrator Rights

DTLknowsWhy works without administrator rights but some data will be missing or less accurate. The following table shows what is affected:

FeatureWithout adminWith admin
System info, network, services, testsFull dataFull data
is_admin fieldFalse — expert.py emits a WARN findingTrue
Some sc query callsMay return incomplete results on hardened machinesFull results

To run as administrator: right-click the command prompt or PowerShell and choose Run as administrator, then run the script from there.

French Windows Note

The network collector parses ipconfig /all output using French-language labels. On English Windows, the following fields will always be None:

  • IPv4 address, subnet mask, default gateway
  • DNS servers
  • DHCP and NetBIOS status
  • Resolved hostname in remote tests

All other data (system info, services, port tests, MAC address, classification) works on any Windows locale.

Known Limitations

  • Windows only. The tool will not run on Linux or macOS.
  • Single active adapter. On machines with multiple network adapters, only the values from the first matching adapter are collected.
  • ARP-based MAC lookup requires LAN adjacency. The target must be on the same Layer 2 segment, or have been recently contacted, for MAC resolution to succeed.
  • Remote tests can be slow. Each TCP port probe uses PowerShell with a 15-second timeout. If all four ports time out, remote tests take up to 60 seconds.
  • Server mode writes files on every request. Each GET /snapshot triggers a full collection and writes new JSON, TXT, and HTML files.
  • Expert and compare messages are French-only. expert.py and compare.py always output in French regardless of --lang.
  • compare.py is directional. Only differences where PC B deviates from PC A are reported. Run expert.py on each snapshot for independent analysis.
  • HTML report logo path is relative. Place netdtl_logo.png in the same directory as the HTML report for the logo to display.
