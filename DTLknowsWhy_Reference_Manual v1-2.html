DTLknowsWhy — Reference Manual v1.2

Complete reference for DTLknowsWhy v1.2 — a Windows network diagnostic agent producing bilingual TXT, HTML and JSON reports.

version1.0 platformWindows 10 / 11 runtimePython 3.10+ languagesfr / en licenseMIT authorDidier DTL Morandi

Overview

DTLknowsWhy is a Python command-line agent for Windows that collects a comprehensive snapshot of the local machine (identity, OS, network configuration, Windows services, local connectivity tests) and optionally runs remote diagnostic tests against a target IP or hostname. It produces three output files per run: a JSON snapshot, a plain-text report, and a self-contained HTML report.

Reports are bilingual: French (--lang fr, default) and English (--lang en). All user-visible strings are managed centrally in shared/i18n.py.

The tool can also expose a minimal HTTP server on port 5050, allowing a remote machine (e.g. running NetDTL) to trigger a snapshot collection via a simple authenticated GET request.

No pip install required: DTLknowsWhy uses only the Python standard library (subprocess, socket, ctypes, getpass, json, logging, http.server). No virtual environment is needed.

Usage

python agent.py [--snapshot] [--target IP_OR_HOST] [--listen] [--lang {fr,en}]
ArgumentDescription
--snapshotCollect local snapshot (system, network, services, local tests) and write output files. Can be combined with --target.
--target HOSTRun remote diagnostic tests against the given IP or hostname, in addition to the local snapshot. Implies snapshot collection.
--listenStart the HTTP server on port 5050 and wait for remote snapshot requests. Mutually exclusive with --snapshot / --target.
--lang {fr,en}Report language. Default: fr.

Examples

# Local snapshot in French (default)
python agent.py --snapshot

# Local snapshot + remote tests, English report
python agent.py --target 192.168.1.42 --lang en

# Start HTTP server (for remote collection from NetDTL)
python agent.py --listen
Tip: Running without any argument prints the help message. --snapshot and --target both trigger create_snapshot(); the difference is that --target also collects remote tests against the specified host.

Output Files

Every run of create_snapshot() writes three files to the current working directory, all named after the local hostname and a timestamp:

FileFormatDescription
HOSTNAME_snapshot_YYYYMMDD_HHMMSS.jsonJSONRaw structured snapshot of all collected data. Suitable for archiving or programmatic processing.
HOSTNAME_report_YYYYMMDD_HHMMSS.txtUTF-8 plain textHuman-readable text report with all sections, formatted with fixed-width labels.
HOSTNAME_report_YYYYMMDD_HHMMSS.htmlSelf-contained HTMLDark-themed HTML report with colour-coded values, collapsible sections for services and local tests, and a target classification badge for remote tests.

All three files are written to the directory from which agent.py is invoked. Errors during collection do not stop the run; missing values appear as None in the snapshot and as Unknown / Inconnu in the reports.

HTML report note: The HTML report references netdtl_logo.png via a relative <img src> path. Place the logo file in the same directory as the report, or the logo space will be empty.

Architecture

Entry point

agent.py
Main entry point. Parses CLI arguments and dispatches to either the HTTP server (--listen) or the snapshot collection function (--snapshot / --target).
Key function

create_snapshot(target=None, lang="fr") — orchestrates the full collection pipeline:

  • Calls collect_system_info(), collect_network_info(), collect_basic_tests(), collect_services()
  • If target is provided, also calls collect_remote_tests(target)
  • Serialises the snapshot to JSON via export_snapshot()
  • Generates and saves the text report via generate_text_report() + save_text_report()
  • Generates and saves the HTML report via generate_html_report() + save_html_report()
  • Returns the full snapshot dict
CLI dispatch

If --listen: imports and calls agent.server.run(). If --snapshot or --target: calls create_snapshot() directly. Without arguments: prints help.

HTTP server

agent/server.py
Minimal HTTP server that exposes a single GET /snapshot endpoint. Allows a remote machine (e.g. NetDTL) to trigger a local snapshot collection and receive the result as JSON over the network.
Configuration constants
ConstantDefaultDescription
API_KEYDTLSECRETPre-shared key sent as ?key= query parameter. Requests with the wrong key receive a 403 response.
HOST0.0.0.0Binds to all network interfaces.
PORT5050TCP port the server listens on.
Endpoint
GET /snapshot?key=DTLSECRET&lang=fr
ParameterRequiredDescription
keyYesMust match API_KEY. Returns 403 if absent or wrong.
langNoReport language, fr (default) or en.
Response

On success: HTTP 200 with Content-Type: application/json; charset=utf-8 and the full snapshot dict as JSON body. On error during collection: HTTP 500 with the exception message.

Side effects

Each remote snapshot request triggers a full create_snapshot() call on the server machine, which writes the three output files locally (JSON, TXT, HTML) in addition to returning the JSON over HTTP.

Security: The API key is stored in plain text in server.py. The server does not use TLS. Use only on trusted local networks.

Collectors

System collector

agent/collectors/system.py
Collects local machine identity and OS information. Uses the Python standard library and a PowerShell registry query for Windows version details.
Output keys
KeySourceDescription
hostnamesocket.gethostname()NetBIOS / DNS hostname of the local machine
usernamegetpass.getuser()Currently logged-in Windows username
is_adminctypes.windll.shell32.IsUserAnAdmin()Boolean — whether the process runs with administrator privileges
windows_product_nameRegistry + normalize_windows_name()Normalised OS name (e.g. Windows 11 Pro). Build ≥ 22000 is classified as Windows 11.
windows_versionRegistry DisplayVersionFeature update version string (e.g. 23H2)
windows_buildRegistry CurrentBuildNumeric build number string (e.g. 22631)
Registry query
HKLM:\SOFTWARE\Microsoft\Windows NT\CurrentVersion
  → ProductName, DisplayVersion, CurrentBuild
Windows 11 normalisation: Windows 11 reports itself as "Windows 10" in ProductName on some builds. normalize_windows_name() corrects this by checking whether the build number is ≥ 22000.

Network collector

agent/collectors/network.py
Collects IPv4 address, subnet mask, default gateway, DNS servers, DHCP and NetBIOS status by parsing ipconfig /all output. Also retrieves the active network profile name and category via PowerShell.
Output keys
KeySourceDescription
active_adapter_profilePowerShell Get-NetConnectionProfileName of the active connection profile
network_categoryPowerShell NetworkCategoryProfile type: Public, Private, or Domain
ipv4ipconfig /allPrimary IPv4 address
subnet_maskipconfig /allSubnet mask
default_gatewayipconfig /allDefault gateway IP
dns_serversipconfig /allList of DNS server IPs (deduped)
dhcp_enabledipconfig /allBoolean — DHCP activated on the primary adapter
netbios_enabledipconfig /allBoolean — NetBIOS over TCP/IP status
Parsing notes

All values are extracted from ipconfig /all stdout via regex. The gateway is extracted with a multi-line pattern (re.DOTALL) to handle the blank-line format used by Windows. DNS servers are collected line by line, stopping at the first blank line after the "Serveurs DNS" label. The parser expects French-language ipconfig output; on English Windows, the regex patterns will not match and values will be None.

French Windows only: The regex patterns in network.py match French-language ipconfig /all labels (Adresse IPv4, Passerelle par défaut, Serveurs DNS, DHCP activé, NetBIOS sur Tcpip). On English Windows, all network fields will be None.

Local tests collector

agent/collectors/tests.py
Runs three local ping tests to verify basic IP stack health: loopback, self (own IPv4 address), and default gateway.
Output keys
KeyTargetDescription
ping_localhost127.0.0.1Loopback test — verifies the IP stack is functional
ping_selfLocal IPv4 from network_infoSelf-ping — verifies the NIC is reachable. Skipped (False) if no IPv4 was collected.
ping_gatewayDefault gateway from network_infoGateway ping — verifies LAN reachability. Skipped (False) if no gateway was collected.

Each test runs ping -n 1 with a 10-second timeout. Returns True if exit code is 0, False otherwise.

Remote tests collector

agent/collectors/remote_tests.py
Runs a battery of remote diagnostic tests against a target IP or hostname: ping, hostname resolution, TCP port probes on ports 80, 139, 443 and 445, MAC address lookup, and automatic target classification.
Output keys
KeyDescription
targetThe IP or hostname passed to collect_remote_tests()
resolved_nameHostname resolved from the IP via ping -a -n 1. None if not resolved or ping failed.
ping_targetBoolean — True if ping -n 1 succeeds (exit code 0)
tcp_80Boolean — TCP port 80 open (PowerShell Test-NetConnection)
tcp_139Boolean — TCP port 139 open (NetBIOS session)
tcp_443Boolean — TCP port 443 open (HTTPS)
tcp_445Boolean — TCP port 445 open (SMB)
mac_addressMAC address from the ARP table (arp -a). None if not found. Only populated if the target responded to ping.
target_typeClassification string — see Target classification
Execution flow

TCP port probes and MAC lookup are only attempted if the initial ping succeeds. The progress bar (progress_bar()) prints scan progress to stdout during port probing. Each Test-NetConnection call has a 15-second timeout.

Hostname resolution

Resolved via ping -a -n 1 TARGET. The hostname is extracted from the response line with the regex \s+sur\s+([^\s]+)\s+\[, which matches the French-language ping format (Envoi d'une requête 'ping' sur HOSTNAME [IP]). Returns None on English Windows.

French Windows only: Hostname resolution from ping output uses a French-language regex. On English Windows, resolved_name will always be None.

Services collector

agent/collectors/services.py
Queries the status of five Windows services critical for network file sharing and name resolution using sc query.
Monitored services
Service nameRole
LanmanServerSMB Server — file and printer sharing
LanmanWorkstationSMB Workstation client — access to remote shares
FDResPubFunction Discovery Resource Publication — network visibility
fdPHostFunction Discovery Provider Host — device discovery

Returned values

Shared Modules

Command runner

shared/commands.py
Thin wrapper around subprocess.run(). Handles encoding detection, timeouts, and exceptions, always returning a consistent dict regardless of outcome.
Function

run_command(command: str, timeout: int = 15) → dict

KeyTypeDescription
stdoutstrDecoded standard output. Empty string on timeout or exception.
stderrstrDecoded standard error.
exit_codeintProcess return code. -1 on timeout, -2 on exception.
Encoding detection

Output bytes are decoded by trying utf-8, cp850, and cp1252 in order. If all fail, decoding falls back to errors="replace". This handles the mix of encodings produced by Windows command-line tools.

All commands are logged at INFO level. Timeouts are logged at ERROR. Exceptions are logged with full traceback via logger.exception().

shared/serializer.py
Writes the snapshot dict to a JSON file in the current working directory.

Writes HOSTNAME_snapshot_YYYYMMDD_HHMMSS.json with indent=2 and ensure_ascii=False. Returns the Path object of the created file.

Text report generator

shared/report.py
Generates the plain-text report as a single string. All section labels use shared/i18n.py for bilingual output. Fixed-width label columns (18 characters) are used for alignment.
Function

Report structure

  • Local machine section: identity, system, network, Windows services, local tests

Helper functions

state(value, lang) — formats a boolean as OK/Closed or OK/Fermé for connectivity test results.

format_timestamp(lang) — formats the current timestamp as DD/MM/YYYY HH:MM:SS (fr) or YYYY-MM-DD HH:MM:SS (en).

HTML report generator

shared/html_report.py
Generates a self-contained HTML report as a string. Uses an embedded dark theme (CSS variables, JetBrains Mono + Rajdhani fonts via Google Fonts) with colour-coded value rendering, collapsible sections, and a target type badge.
Function

generate_html_report(snapshot: dict, lang: str = "fr") → str

Report structure
  • Header bar with title, generation timestamp, and logo (netdtl_logo.png)
  • Local machine section: identity, system, network, Windows services (collapsible <details>), local tests (collapsible)
  • Remote target section (only if snapshot["remote_tests"] is present): identification with target type badge, remote connectivity tests
Helper functions

yn(value, lang) — returns a <span> with class val-yes, val-no, or val-unknown.

state(value, lang) — returns a <span> with class val-ok, val-closed, or val-unknown.

badge_class(target_type) — maps a classification string to a CSS badge class: green (Windows), blue (mobile Apple/Android), orange (probable device / unknown device), red (unreachable), gray (default).

format_timestamp(lang) — same as in report.py.

MAC address display

If the remote target is the local machine's own IP (self-test), the MAC field displays a localised explanatory string instead of a MAC address, since ARP cannot resolve the local machine's own MAC in this context.

Internationalisation

shared/i18n.py
Central store for all user-visible strings in French and English. Every label, badge text, and interpretation paragraph in both the text and HTML reports is looked up here.
Function

tr(key: str, lang: str = "fr") → str

Returns the translation of key for lang. Falls back to French if the language is not found. Returns the key itself if no translation exists for it.

String categories
  • Section headings: local_machine, remote_target, identity, system, network, windows_services, local_tests, identification, remote_tests, interpretation
  • Field labels: hostname, username, administrator, operating_system, build, profile, ip_address, subnet_mask, gateway, dns_servers, dhcp, netbios, target_ip, resolved_name, target_type, mac_address, ping

Adding a language: add a new key to the top-level TRANSLATIONS dict with all required keys, then pass the new language code via --lang.

Logger

shared/logger.py
Sets up the application-wide Python logger. Writes to dtlknowswhy.log in the current working directory at INFO level.
Usage

Import the ready-to-use singleton: from shared.logger import logger. The logger is named DTLknowsWhy and uses the format %(asctime)s [%(levelname)s] %(message)s. Handlers are only attached once (idempotent setup).

Every run_command() call logs the full command string at INFO level. Timeouts and exceptions are logged at ERROR level with tracebacks.

Expert Modules

Expert diagnosis

expert.py
Entry point for the expert analysis engine. Loads a snapshot JSON file, runs it through the rules engine, and prints a structured diagnostic report to stdout. If no snapshot is specified, automatically picks the most recently modified snapshot file in the current directory.
Usage
python expert.py [snapshot.json]
ArgumentDescription
snapshotOptional. Path to a snapshot JSON file. If omitted, the most recently modified *_snapshot_*.json file in the current directory is used automatically.
Output format

Each finding is printed as [LEVEL] message followed (when applicable) by an indented remediation line:

[FAIL] Le service LanmanServer (partage Windows) est arrêté.
      Action : Démarrer le service : sc start LanmanServer

[OK] Connectivité locale correcte (passerelle 192.168.1.1).

[WARN] Le profil réseau Public peut bloquer le partage Windows.
      Action : Passer le profil réseau en Privé.
Snapshot auto-detection

find_latest_snapshot() globs *_snapshot_*.json in the current directory, sorts by modification time descending, and returns the first result. Returns None if no matching file is found, which causes the script to exit with code 1.

Report header

Before the findings, the report prints: machine hostname, Windows product name and version, network profile, local IP, and remote target (if present in the snapshot).

Comparative analyser

compare.py
Compares two snapshots from two different machines (PC A and PC B) to identify configuration differences that could explain why a feature works on one machine and not the other. Outputs a causal diagnosis to stdout.
Usage
python compare.py snapshot_a.json snapshot_b.json
ArgumentDescription
snapshot_aPath to the reference snapshot (the machine that works correctly).
snapshot_bPath to the snapshot of the machine under investigation.
Comparison rules
TagConditionMessage
[CAUSE PROBABLE]PC B network_category == "Public" while PC A differsPublic network profile may block SMB and network discovery
[DONNÉES INSUFFISANTES]PC B network_category is None, "Unknown", or "(unknown)" while PC A differsNetwork profile of PC B is unknown
[CAUSE CERTAINE]PC B LanmanServer == "Stopped" while PC A differsWindows file sharing disabled on PC B
[CAUSE CERTAINE]PC B LanmanWorkstation == "Stopped" while PC A differsSMB client stopped on PC B
[CAUSE PROBABLE]PC A ping_gateway is True and PC B ping_gateway is FalsePC B has a local connectivity problem (gateway unreachable)
[AUCUNE ANOMALIE MAJEURE]No findings generatedNo critical difference detected
Output format

Plain text, printed directly to stdout. No files are written. Each finding is preceded by its tag on a separate line.

Convention: PC A is the reference machine (working correctly). PC B is the machine being diagnosed. The comparison is directional — only B's deviations from A are flagged.

Target Classification

After remote tests complete, classify_target() in remote_tests.py assigns a target_type string based on the test results. The rules are applied in order; the first match wins:

target_typeConditionBadge colour
probable_mobile_appleResolved hostname contains iphone or ipadBlue
probable_mobile_androidResolved hostname contains androidBlue
probable_windowsTCP 445 or TCP 139 openGreen
probable_deviceTCP 80 or TCP 443 openOrange
unknown_network_deviceMAC address resolved but no open portsOrange
unknown_hostPing succeeded but no MAC and no open portsGrey
unreachablePing failedRed
Note: hostname-based classification (Apple / Android) takes priority over port-based classification. A device with TCP 445 open whose hostname contains iphone will be classified as probable_mobile_apple, not probable_windows.

Known Limitations

  • Windows only. The tool imports ctypes.windll, calls ipconfig, sc query, arp -a, tracert, and PowerShell. It will not run on Linux or macOS.
  • French Windows assumed for network parsing. The regex patterns in network.py and the hostname resolution pattern in remote_tests.py match French-language Windows output. On English Windows, all network fields and the resolved hostname will be None.
  • Single active adapter. network.py extracts values from the first matching adapter in ipconfig /all output. On machines with multiple adapters, only the first match for each field is returned.
  • ARP-based MAC lookup requires LAN adjacency. get_mac_address() reads the ARP table. The target must be on the same Layer 2 segment (or a recent ARP cache entry must exist) for MAC resolution to succeed.
  • Self-test MAC not available. When the remote target is the local machine's own IP, arp -a does not return the machine's own MAC. The report displays an explanatory message instead.
  • Port probes use Test-NetConnection. Each TCP port check invokes PowerShell, which adds startup overhead. The 15-second timeout per port means remote tests can take up to 60 seconds if all four ports time out.
  • No persistent server state. The HTTP server (--listen) holds no state between requests. Each GET /snapshot triggers a full collection and writes new output files.
  • API key in plain text. API_KEY is hardcoded in server.py. There is no TLS. The server is intended for use on trusted local networks only.
  • Log file location. dtlknowswhy.log is written to the current working directory. If the process lacks write permission, logging will silently fail.
  • Comparative analysis is directional. compare.py treats PC A as the reference. Only differences where PC B deviates from PC A are reported. Reverse comparison requires swapping the arguments.
  • Expert engine messages are French-only. rules_engine.py and compare.py output messages in French exclusively, unlike the main reports which support --lang en.
